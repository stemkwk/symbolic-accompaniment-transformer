# 🗂️ 전조 증강의 오프라인 사전 빌드 (Offline Pitch Augmentation) - 기각 (Rejected)

본 문서에서는 GPU 서버의 데이터 공급 속도를 극대화하고 CPU 연산 비용을 줄이기 위해 전조 증강(Pitch Transposition)을 오프라인으로 이전하는 방안의 검토 및 최종 기각 사유를 기술합니다.

---

## ❌ 기각 요약

* **제안 내용**: 모든 음높이 전조($_12$개 조표) 결과물을 오프라인 전처리 단계에서 미리 `.pt` 파일로 빌드해놓음으로써 학습 중 CPU 부하를 없애자는 제안.
* **최종 판정**: **기각 (Rejected)**
* **기각 핵심 사유**:
  1. **실시간 텐서 벡터화 연산 도입 완료**: `dataset.py` 내의 전조 루프를 PyTorch 텐서 벡터화($O(1)$) 연산으로 성공적으로 리팩토링하여, 학습 중 Dynamic하게 전조하더라도 CPU 병목 및 GPU Starvation이 완전히 사라졌습니다.
  2. **저장 용량 비효율**: 12개 전조를 사전 생성할 경우 디스크 용량이 불필요하게 비대해지며(최대 1.8GB+), 도커 컨테이너 빌드 및 네트워크 전송 효율을 떨어뜨립니다.
  3. **Jitter 연산과의 조화**: Velocity, Duration, Tempo Jitter는 음표별 독립 적용으로 인해 사전 연산이 불가능하므로, 어차피 dynamic data augmentation 파이프라인이 유지되어야 합니다.

## 🔍 상세 기각 사유

### 1. 실시간 텐서 벡터화 최적화 성공
이전에는 CPU 단에서 Python 단일 스레드로 `for` 루프를 돌며 음표를 하나씩 조옮김하느라 GPU가 노는 현상이 있어 오프라인 빌드가 대안으로 제시되었습니다. 하지만 [performance_optimization.md](file:///c:/Users/hojun/Documents/대학교 자료/3학년 1학기(2026-1)/기학지/과제/project_transformer/analysis/performance_optimization.md)에서 기술된 것처럼 PyTorch 벡터 연산으로 리팩토링이 완료되어, CPU 단의 연산 레이턴시가 마이크로초(μs) 단위로 떨어졌습니다. 따라서 오프라인 빌드로 인한 디스크 이점 대비 구현 및 자원 낭비가 큽니다.

### 2. 디스크 용량의 불필요한 비대화
상대적 화성 토크나이저 덕분에 12개 조표를 전량 빌드할 필요성이 매우 낮습니다. 절대 음높이가 바뀌어도 Key-relative 표현식은 완전히 일체화되므로, 단순히 옥타브 경계선 일반화와 전역 KEY 지정을 위해서만 전조가 동작합니다. 오프라인으로 12개 피치를 전부 복사하는 것은 네트워크 대역폭과 디스크 I/O 자원의 낭비입니다.

### 3. Jitter의 실시간(Dynamic) 유지 필요성과의 충돌
Velocity / Duration / Tempo Jitter 연산은 매 에폭마다 다르게 흔들려야 과적합을 제대로 방지합니다. 음표별로 독립적인 노이즈를 입히는 조합의 수($5^{\text{음표수}}$)는 디스크에 사전 생성하는 것이 수학적으로 불가능합니다. 어차피 배치 생성 시 실시간 데이터 증강(Dynamic Augmentation) 코드가 돌아가야 하므로, 전조 연산 역시 이 파이프라인 내에서 효율적인 텐서 연산으로 함께 수행되는 것이 설계상 간결하고 강력합니다.
