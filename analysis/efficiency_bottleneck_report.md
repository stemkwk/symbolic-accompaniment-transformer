# JAM Transformer 효율성 및 병목(Bottleneck) 분석 리포트

본 프로젝트는 구조적으로 매우 고도화되어 있으나, 대규모 데이터셋을 학습할 때 치명적으로 작용할 수 있는 시스템 메모리(RAM) 병목이 존재합니다. 학습 서버 배포 전 반드시 확인해야 할 장단점과 대처 방안을 정리했습니다.

---

## 1. 🟢 고도로 최적화된 영역 (Highly Efficient)

프로젝트 전반에 걸쳐 하드웨어 자원을 극한으로 활용하는 최신 최적화 기법들이 적용되어 있습니다.

### 1.1. GPU 메모리(VRAM) 및 연산 최적화
* **Flash Attention 지원**: PyTorch의 `scaled_dot_product_attention`을 활용하여, 지원되는 하드웨어(SM 8.0 이상)에서 $O(N^2)$의 어텐션 메모리 폭발을 방지합니다. 구형 하드웨어 감지 시 논리적 배치(Gradient Accumulation)를 통해 자동으로 VRAM을 방어하는 로직이 훌륭합니다.
* **KV-Cache (추론 속도 향상)**: `model.py`에 적용된 KV-Cache 덕분에 매 토큰 생성 시 이전 상태를 다시 계산할 필요가 없어 웹 데모(UI)의 지연 시간(Latency)이 획기적으로 단축되었습니다.
* **`torch.compile` 호환성 확보**: 위치 인코딩(RoPE) 버퍼를 텐서 뷰(Slicing)로만 접근하도록 설계하여, PyTorch 2.0+ 컴파일러의 CUDA Graph 생성 시 주소 재할당 에러를 방지했습니다.

### 1.2. 데이터 증강 (CPU Vectorization)
* 느린 Python `for` 루프 대신 텐서 슬라이싱(`ids[:-1]`, `ids[1:]`)을 이용한 벡터화 연산(Vectorized Operations)을 통해 피치(Pitch) 이동을 처리하여 CPU 병목을 제거했습니다.

## 2. 🔴 치명적인 병목 및 위험 요소 (System RAM OOM) — ✅ 해결됨 (2026-05-31)

> **해결 완료** (커밋 `5f3b482`): 아래 무한 증식 캐시를 **RAM 티어 기반 LRU 상한 캐시**로 교체했습니다.
> `config.env_scaling.ram_tiers`(≥64→24GB / ≥32→10 / ≥16→4 / else→2)로 총 예산을 정하고
> `get_worker_info()`로 워커 수만큼 분할해, 워커 합산 RAM이 티어를 넘지 않도록 LRU로 evict합니다.
> 18,161 shard(≈2.8GB) 기준 16GB 박스는 전부 캐시(손실 0), Colab·다워커는 자동 상한. 아래는 원래 분석.

가장 큰 위험 요소는 **System RAM(메모리) 폭발로 인한 학습 중단(OOM)** 가능성입니다.

### 2.1. 원인 분석 (`dataset.py` 의 `_shard_cache`)
```python
if shard_idx not in self._shard_cache:
    data = torch.load(...)
    self._shard_cache[shard_idx] = data  # 한 번 읽은 파일을 영구 저장
```
* **캐시 무한 증식**: 디스크 I/O 속도를 높이기 위해, 데이터를 한 번 읽어오면 `_shard_cache` 딕셔너리에 영구적으로 보관합니다. 수백 곡 단위에서는 매우 빠르지만, **18,000곡 이상의 대용량 데이터셋**에서는 수십 GB의 데이터가 메모리에 무한정 쌓이게 됩니다.
* **멀티 프로세싱 증폭**: PyTorch의 `DataLoader`는 `num_workers > 0`일 때 여러 개의 백그라운드 프로세스를 생성합니다. 각 프로세스가 이 거대한 `_shard_cache`를 독립적으로 복제하여 램에 쌓게 되므로, RAM 사용량이 워커 수(4배, 8배)만큼 폭발적으로 증폭됩니다.

## 3. 💡 해결책 및 권장 사항

현재 코드를 유지한 상태로 대규모 학습을 시도하려면 다음 중 하나의 조치를 반드시 취해야 합니다.

1. **고용량 RAM 인스턴스 확보**
   * 클라우드 서버 렌탈 시, GPU VRAM뿐만 아니라 **System RAM이 최소 32GB~64GB 이상**인 환경을 선택해야 합니다. (Colab 무료 버전의 12GB RAM 환경에서는 1 Epoch를 돌기도 전에 세션이 다운될 확률이 100%입니다.)
2. **워커(Worker) 수 제한**
   * RAM이 넉넉하지 않은 환경이라면, `config.yaml`의 `num_workers` 값을 `0`으로 설정하여 캐시 복제를 막아야 합니다.
3. **(옵션) 캐시 기능 비활성화**
   * 대규모 학습 시 디스크 I/O를 약간 희생하더라도 메모리 터짐을 방지하려면, `dataset.py`의 `_shard_cache` 딕셔너리 할당 부분을 주석 처리하고 매번 `torch.load`를 통해 디스크에서 직접 읽어오도록 수정하는 것이 가장 안전합니다.

## 4. 🚀 프로덕션 레벨의 심화 최적화 분석 (`pipeline.py`)

초기 분석 시 `model.py` 단독 사용 시 발생할 수 있는 병목(텐서 재할당)을 우려했으나, 실제 앱(Web UI)을 구동하는 핵심 엔진인 `pipeline.py`를 정밀 분석한 결과 **최신 LLM 서빙 프레임워크 수준의 고도화된 기법들로 모든 병목이 이미 우회 및 해결되어 있음**을 확인했습니다.

### 4.1. O(1) 리스트 Append를 통한 텐서 재할당 병목 완벽 회피
* PyTorch에서 매 토큰 생성 시 텐서의 길이를 늘리는(`torch.cat`) 작업은 극심한 메모리 단편화를 유발합니다. 
* 하지만 `pipeline.py`의 `generate_accompaniment` 함수는 무거운 텐서 병합을 피하고 파이썬의 `list.append()`를 활용해 $O(1)$ 속도로 토큰 ID만 수집한 뒤, 모델에는 단일 토큰(`[next_id]`)만 전달하도록 설계되어 있어 **추론 속도 지연(Latency Bottleneck)이 원천 차단**되어 있습니다.

### 4.2. 무한 길이 생성을 위한 "슬라이딩 윈도우 (Sliding Window)" KV-Cache 관리
* 트랜스포머는 학습된 컨텍스트 길이를 초과하면 VRAM OOM이 발생하거나 모델의 추론력이 붕괴됩니다.
* 본 프로젝트는 캐시가 한계에 다다르면 가장 앞의 글로벌 정보(BOS, 조표, 템포)만 남기고 오래된 중간 기억을 잘라내는 `_maybe_slide` 함수 로직을 구현했습니다. 
* 이는 최근 학계에서 주목받은 **StreamingLLM (Attention Sink)** 논문의 핵심 아이디어와 정확히 일치하며, 이를 통해 곡의 길이에 상관없이 무한정 잼(Jam) 연주가 가능합니다.

### 4.3. 화성학 지식을 결합한 "제약 기반 디코딩 (Constrained Decoding)"
* 데이터 기반의 딥러닝 모델이 낼 수 있는 '불협화음'의 한계를 **음악적 도메인 지식**으로 극복했습니다.
* 연주 중인 코드(Chord)를 실시간으로 추적하여 부딪히는 기피음(Avoid note)의 확률(Logit)을 능동적으로 깎아내는(Soft Penalty) 로직을 통해, 단순한 기계학습 모델이 아닌 화성학적 룰이 결합된 안정적인 AI 작곡가 시스템을 완성해 냈습니다.
